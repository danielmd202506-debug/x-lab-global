# X-LAB 官网 UI/UX 重构项目计划

版本：1.0  
日期：2026-06-06  
用途：指导 X-LAB 官网后续 UI 设计、UX 梳理、页面重构、交互实现和验收。

## 1. 项目定位

X-LAB 官网不是单一品牌展示页，而是一个面向全球市场的全链路品牌门户。它需要同时服务品牌信任、工厂能力、供应商合作、经销商合作、门店支持和骑手购买信心。

本次 UI/UX 重构的目标是让访客在进入网站后快速理解三件事：

- X-LAB 是一个具备 race validation、factory control、channel readiness 的自行车品牌。
- 不同角色有明确入口：supplier、dealer、store、rider、B2B partner。
- 每个页面都能给出业务证据，并引导用户进入下一步操作。

## 2. 项目目标

### 2.1 品牌目标

- 建立技术、直接、高压、工业化的视觉气质。
- 用真实产品、工厂、赛事、门店和业务流程图像替代抽象装饰。
- 强化 X-LAB 的核心叙事：Race proven, factory controlled, channel ready。

### 2.2 用户体验目标

- 让不同用户不用猜入口，能直接进入对应路径。
- 减少泛泛介绍，增加可操作模块，例如配置、询价、登录、合作申请、门店查询。
- 把“证明”放在“转化”前面：先展示能力、质量、渠道和服务，再引导询盘或合作。

### 2.3 转化目标

- 提升 B2B 登录、经销商合作、供应商入口、车型配置和询价路径的可见性。
- 让产品发现、配置、RFQ、合作申请形成连续路径。
- 为后续数据追踪准备明确的 CTA、页面目标和事件节点。

## 3. 当前设计基准

后续设计必须优先继承当前项目已有的品牌系统。

### 3.1 品牌视觉方向

- 技术感、工业感、竞速感、业务导向。
- 深色工业背景作为 public brand 页面主基调。
- 浅色工作台式界面用于 B2B dashboard、表单、运营数据和任务处理。
- 页面结构必须体现上游到下游链路：brand、factory、supplier、dealer、store、rider。

- Logo：`assets/x-lab-logo.webp`
- 全局样式：`styles.css`

### 3.3 已有页面基础

## 5. 项目阶段与活动

### 阶段 1：诊断与策略确认

目标：明确重构范围、问题优先级和设计方向。

活动：

- 审查现有页面结构、导航、CTA、内容密度和移动端体验。
- 梳理不同用户角色的进入路径和退出路径。
- 标记当前页面中的重复内容、断裂路径和缺失证据。
- 确认本期必须完成的页面和可延期的模块。
- 输出页面优先级和重构范围清单。

交付物：

- 官网现状诊断清单
- 页面优先级表
- 用户路径草图
- 重构范围说明

### 阶段 2：信息架构与用户路径

目标：建立清晰、可扩展的官网结构。

活动：

- 重新定义主导航和页脚结构。
- 设计角色入口：Supplier、Dealer、Store、Rider、B2B。
- 规划每个页面的上一步、当前任务、下一步。
- 明确每个页面需要展示的业务证据。
- 设计核心转化路径：Configure Bike、Create RFQ、Become a Dealer、Supplier Login、B2B Login、Find Store。

交付物：

- Sitemap
- 用户路径图
- 页面目标矩阵
- CTA 路径矩阵

### 阶段 3：内容策略与页面叙事

目标：让每个页面都有明确角色、证据和动作。

活动：

- 重写首页品牌叙事，让用户快速理解 X-LAB 全链路定位。
- 为工厂、供应商、经销商、门店、骑手分别制定内容层级。
- 建立页面内容模板：角色定位、能力证明、业务流程、下一步动作。
- 精简卡片文案，避免空泛 premium、innovation、future 等泛词。
- 为按钮和链接制定明确动词，例如 Request allocation、Create RFQ、Open supplier login、Build bike。

交付物：

- 页面内容大纲
- 核心文案方向
- CTA 文案表
- 业务证据清单

### 阶段 4：视觉系统强化

目标：在现有设计系统基础上强化官网统一性和识别度。

活动：

- 复核当前色彩 token 的使用规则。
- 明确红色、青色、金色、绿色在品牌和运营界面中的语义。
- 规范 hero、section head、process strip、role link、product card、dashboard card 的使用场景。
- 检查字体层级，保证移动端大标题不溢出。
- 确认图像策略：hero 使用真实环境，产品卡使用可识别产品，运营模块使用业务上下文。
- 补充必要的组件状态：hover、focus、active、disabled、empty、loading、error、success。

交付物：

- UI 视觉方向板
- 组件清单
- 页面布局规范
- 视觉验收 checklist
- `design.md` 更新建议

### 阶段 5：关键页面 UI 设计

目标：优先完成影响品牌理解和转化路径的核心页面。

| P1 | Factory | 生产能力、质量门、可追溯性 |

活动：

- 为 P0 页面先做完整 UI 方向。
- 使用真实内容密度进行设计，不用 lorem ipsum。
- 同时设计桌面端和移动端关键断点。
- 检查 CTA 是否和页面目标一致。
- 记录新的可复用组件，并同步到设计系统。

交付物：

- P0 页面高保真设计
- P1 页面设计方案

### 阶段 6：交互与功能设计

目标：让官网从展示页面变成可用业务入口。

活动：

- 设计配置器流程：选择车型、尺码、颜色、套件、轮组、配件、保存、询价。
- 设计 RFQ 流程：产品、数量、交期、备注、联系方式、提交反馈。
- 设计经销商申请流程：地区、公司信息、销售规模、合作意向、提交状态。
- 设计供应商登录和资料提交流程。
- 设计 B2B dashboard 的筛选、表格、状态、提醒和角色切换。
- 设计错误、空状态、加载、提交成功和权限不足状态。

交付物：

- 交互流程图
- 表单字段清单
- 状态设计清单
- Dashboard 操作模型
- 原型或可点击流程

### 阶段 7：前端实现与设计落地

目标：把设计系统、页面结构和交互转化为稳定页面。

活动：

- 复用现有 `styles.css` token、布局、按钮、卡片、表单和 dashboard 样式。
- 新增组件前先确认是否能由现有系统表达。
- 实现响应式布局，重点检查 980px 和 640px 断点。
- 保持可访问性：焦点状态、语义标签、表单 label、对比度、减少动态效果。
- 优化图片尺寸、加载表现和首屏稳定性。
- 更新 `design.md`，记录任何跨页面设计变化。

交付物：

- 完成的 HTML/CSS/JS 页面
- 更新后的设计系统文档
- 响应式检查记录
- 可访问性检查记录

### 阶段 8：测试、验收与上线准备

目标：确保页面能真实使用，而不是只在设计稿中好看。

活动：

- 检查桌面、平板、移动端页面无布局错位和文字溢出。
- 测试所有 CTA 链接是否通向正确页面。
- 测试配置器、登录、表单、RFQ、dashboard 状态。
- 检查图片是否加载，是否表达产品、工厂、门店或业务上下文。
- 检查浏览器兼容：Chrome、Edge、移动端 Safari。
- 准备上线前内容、SEO title、meta description 和统计事件。

交付物：

- QA 检查表
- 修复清单
- 上线 checklist
- 数据追踪事件表

## 6. 页面设计指导

### 6.1 首页

首页必须完成四个任务：

- 第一屏明确 X-LAB 的品牌定位。
- 展示从 factory 到 supplier、dealer、store、rider 的业务链路。
- 给出产品和配置入口。
- 保留 B2B Login 作为稳定商业入口。

建议模块：

- Hero：race road 与 factory credibility 结合。
- Chain Strip：Brand / Factory / Supplier / Dealer / Store / Rider。
- Proof Section：比赛、制造、QC、交付能力。
- Product Gateway：主车型或重点车型入口。
- Partner Routes：supplier、dealer、store 分路径。
- Final CTA：Build bike / Open B2B login / Become a dealer。

### 6.2 产品与配置页面

设计重点：

- 产品图像必须可识别，不做纯氛围背景。
- 配置流程要像工具，不像宣传页。
- 每一步都要显示当前选择、兼容性和下一步动作。
- RFQ 或保存配置要始终清晰可见。

关键状态：

- 未选择
- 已选择
- 不兼容
- 推荐
- 已保存
- 可提交 RFQ

### 6.3 经销商页面

设计重点：

- 先证明品牌、产品、供货、售后，再要求合作申请。
- 展示合作利益时避免空泛口号，要具体到 allocation、sell-in、warranty、training、stock planning。
- 页面 CTA 应该是 Become a dealer、Request allocation、Open B2B login。

### 6.4 供应商页面

设计重点：

- 供应商更关心标准、文件、交付、质量门和合作流程。
- 页面应表现为严肃的资格与协作入口，不是品牌宣传页。
- CTA 应该是 Open supplier login、Submit qualification、View material requests。

### 6.5 B2B Dashboard

设计重点：

- Dashboard 要安静、密集、可扫描。
- 不使用大幅品牌 hero。
- 优先表格、筛选、KPI、状态提醒和任务流。
- 设计应支持重复操作和快速对比。

## 7. 设计原则

后续 UI 设计必须遵守以下原则：

- 每个页面先定义用户角色，再设计版式。
- 每个 CTA 必须对应真实下一步，不使用泛泛 Learn more。
- 每个页面都要有业务证据，不能只有品牌口号。
- 卡片只用于真实重复内容、产品、任务或表单，不把整页做成卡片堆叠。
- 颜色保持语义稳定：红色主动作，青色技术信号，金色警告，绿色成功。
- Hero 和产品区域必须使用真实或可信图像。
- 移动端优先检查导航、标题换行、表单和 CTA 可点击性。
- Dashboard 以效率为先，官网 public 页面以品牌和路径清晰为先。

## 10. 验收标准

### 10.1 体验验收

- 用户能在 5 秒内判断自己应该进入哪个路径。
- 首页能解释 X-LAB 的品牌、工厂、渠道和购买路径。
- 每个关键页面至少有一个清晰主 CTA。
- 角色路径之间不混淆，supplier、dealer、store、rider 入口清楚。

### 10.2 视觉验收

- 页面符合当前 X-LAB 设计系统。
- 色彩、字体、按钮、卡片、表单和导航保持一致。
- 图像能表达产品、工厂、门店、路线或业务上下文。
- 移动端无文字溢出、按钮拥挤或内容遮挡。

### 10.3 功能验收

- 所有页面链接可访问。
- 表单和配置流程有完整状态反馈。
- Dashboard 能完成筛选、查看、切换和操作反馈。
- B2B 和 supplier 登录路径明确。

### 10.4 可访问性验收

- 文本对比度达到 WCAG AA 目标。
- 键盘焦点清晰可见。
- 表单 label 可见。
- 动效支持 `prefers-reduced-motion`。
- 链接和按钮文案可独立理解。
